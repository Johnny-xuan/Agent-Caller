# Supervised Execution

## Public Policy

Agent Caller keeps resource scope and approval behavior independent:

```text
sandbox:  read_only | workspace_write | danger_full_access
approval: fail_closed | on_request | autonomous
```

Named profiles provide the normal product interface:

```text
trusted  = danger_full_access + autonomous
guarded  = workspace_write   + on_request
observer = read_only         + fail_closed
```

New agents default to `trusted` when no profile or low-level policy is supplied.
The default is intentional for this owner's personal local plugin: external
agents should behave like trusted native sub-agents during ordinary coding work.

The low-level defaults remain asymmetric for backward-compatible calls that
pass `sandbox` or `approval` directly:

- `read_only` defaults to `fail_closed`.
- `workspace_write` defaults to `on_request`.
- `danger_full_access` defaults to `on_request` and cannot be combined with
  `fail_closed`, because no sandbox remains to fail closed against.

The parent Codex task's current policy is not inherited. Standard MCP calls do
not send the host sandbox or approval policy to the MCP server. The profile is
therefore selected by Agent Caller rather than copied from the host task.

The policy selected at `create_agent` is a ceiling. `send_message` may narrow it
for one Run but cannot widen it. Sandbox order is `read_only < workspace_write <
danger_full_access`; approval order is `fail_closed < on_request < autonomous`.

Both providers receive the same delegation prompt. In trusted mode it tells the
agent to proceed without routine local-work approvals while strictly limiting
it to the delegated task and requiring explicit authorization for unrelated
scope, secrets, publication, deployment, destructive Git operations, broad
deletion, and other irreversible or externally visible effects. This prompt is
a soft behavioral constraint. The provider sandbox remains the hard resource
boundary in guarded and observer modes.

## Provider Mapping

| Public policy | Claude Code | Codex |
|---|---|---|
| `read_only` | Read, Glob, Grep, and questions only; write and Bash tools unavailable | read-only sandbox |
| `workspace_write` | Claude command sandbox enabled, unsandboxed commands disabled, workspace added as writable | workspace-write sandbox with the agent cwd as writable root |
| `danger_full_access` | no Claude command sandbox | danger-full-access sandbox policy |
| `fail_closed` | `dontAsk`; unapproved tool requests are denied | `never`; sandbox escalation fails instead of prompting |
| `on_request` | `default` plus SDK callbacks | `on-request` plus App Server request/response |
| `autonomous` | `bypassPermissions` with an explicit dangerous opt-in; the selected command sandbox still applies when present | `never`; the selected sandbox remains in force |

The mappings preserve product meaning but are not identical security engines.
Claude combines tool availability, permission rules, and its command sandbox.
Codex applies its native sandbox policy. Agent Caller must describe the selected
scope honestly rather than claiming provider-independent OS isolation.

Claude CLI fallback does not support interactive callbacks. An agent configured
with `approval=on_request` therefore requires the Claude Agent SDK and fails
clearly instead of silently denying or replaying work. Codex uses App Server v2;
the adapter is verified against the protocol generated by local Codex CLI
0.142.5.

## Request Lifecycle

Requests are stored under the owning Agent and Run:

```text
agents/<agent-id>/requests/<request-id>.json
```

A request contains safe coordinator-facing data and internal provider routing:

```json
{
  "id": "req_...",
  "agentId": "agt_...",
  "runId": "run_...",
  "provider": "claude-code",
  "kind": "tool_approval",
  "status": "pending",
  "title": "Bash requires approval",
  "tool": "Bash",
  "input": { "command": "npm test" },
  "questions": null,
  "availableDecisions": ["allow_once", "allow_session", "deny", "cancel"]
}
```

Provider request IDs are persisted for routing but removed from MCP results.
Secret answers are delivered in memory to the active provider callback and are
stored as `[REDACTED]`.

The active lifecycle is:

```text
running
  -> waiting_for_input
  -> respond_to_request
  -> running
  -> completed | waiting_for_input | failed | stopped
```

`send_message(wait=true)` and `respond_to_request(wait=true)` return at either a
terminal Run state or the next pending request. They do not hold an opaque MCP
call open while a decision is needed.

If Agent Caller stops, the provider callback disappears. Startup recovery marks
the Run interrupted and every pending request expired. A request owned by a
different still-running Agent Caller process is visible but cannot be answered
from the new process; it must be handled by its owner or stopped after the owner
exits. Interrupted write work is never replayed automatically.

## Supported Requests

Claude Agent SDK:

- tool permission through `canUseTool`;
- `AskUserQuestion` answers;
- MCP elicitation through `onElicitation`;
- declared user dialogs through `onUserDialog`.

Codex App Server:

- command execution approval;
- file-change approval;
- additional permission approval;
- `request_user_input` questions;
- MCP form and URL elicitation.

Unknown provider request types fail explicitly. Agent Caller never guesses an
approval response for a request it cannot represent.
